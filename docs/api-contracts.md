# API Contracts

HTTP API implemented by the **Django** app under `backend/`. The **Next.js** app in `cognito-ai/` calls these as same-origin `/api/...` paths; `next.config.ts` rewrites them to `NEXT_PUBLIC_BACKEND_URL` (see `docs/backend-infra.md`).

The AI layer is an **OpenRouter tool-calling skill runner** (`backend/agent/runner.py`) that loads specs from `backend/skills/*.py` — not a separate “Nanoclaw” service.

## Frontend integration (actual behavior)

- **Lessons:** `GET /api/lessons/{lesson_id}` accepts query params `module_topic` and `mode` (`learn` | `solve`); see `lib/lessons.ts` and `backend/apps/lessons/views.py`.
- **Authorization:** The web client attaches `Authorization: Bearer <token>` using values from `lib/auth.ts`. After Firebase sign-in, the stored token is currently the **Firebase user id**, not necessarily the `session_token` returned by the auth endpoints below. Backend stub settings may accept this in dev; production should validate Firebase ID tokens or use a proper session exchange.

## Not implemented yet

- None — all endpoints below are implemented (Sprint 5 complete).

## Auth

### POST /api/auth/register

- Purpose: create a local Django user with email/password and return a session.
- Request:

```
{
  "email": "string",
  "password": "string",
  "name": "string"
}
```

- Response:

```
{
  "session_token": "string",
  "user": {
    "id": "string",
    "email": "string",
    "name": "string"
  }
}
```

### POST /api/auth/login

- Purpose: exchange email/password for backend session.
- Request:

```
{
  "email": "string",
  "password": "string"
}
```

- Response:

```
{
  "session_token": "string",
  "user": {
    "id": "string",
    "email": "string",
    "name": "string"
  }
}
```

### POST /api/auth/firebase-login

- Purpose: exchange Firebase ID token for backend session.
- Request:

```
{
  "id_token": "string"
}
```

- Response:

```
{
  "session_token": "string",
  "user": {
    "id": "string",
    "email": "string",
    "name": "string"
  }
}
```

## Roadmaps

### GET /api/roadmaps

- Returns roadmaps for the authenticated user. Curriculums are returned separately from `GET /api/curriculums`.

### POST /api/roadmaps

- Request:

```
{
  "topic": "string",
  "mode": "learn|solve",     // optional, defaults to "learn"
  "force_roadmap": false      // optional, default false. When true, skip the
                              // broad-topic curriculum split and always return
                              // a roadmap.
}
```

- Response is **discriminated** by `kind`:

```
// Narrow topic — a roadmap is created and persisted (status 201)
{
  "kind": "roadmap",
  "roadmap": {
    "roadmap_id": "string",       // UUID4, generated by backend
    "id": "string",
    "topic": "string",
    "mode": "learn|solve",
    "type": "topic|problem",
    "created_at": "ISO 8601",
    "modules": [
      // learn mode: 1–7 modules; solve mode: 1–5 modules
      { "id": "string", "title": "string", "index": 0, "outcome": "string" }
    ],
    "progress": 0
  }
}
```

```
// Broad topic — a non-persisting curriculum preview (status 200)
{
  "kind": "curriculum_preview",
  "plan": {
    "topic": "string",
    "mode": "learn",
    "courses": [                        // 2–6 courses
      { "id": "string", "index": 0, "title": "string", "outcome": "string" }
    ],
    "source": "agent|placeholder"
  }
}
```

The frontend should ask the user to confirm a `curriculum_preview` plan before
calling `POST /api/curriculums`. If the user declines, send `force_roadmap: true`
on a follow-up `POST /api/roadmaps` to bypass the split.

### GET /api/roadmaps/{roadmap_id}

- Returns roadmap detail with modules and progress.

## Curriculums

A `Curriculum` groups 2–6 `CurriculumCourse` rows; each course can have its own
`Roadmap` once expanded. The first course of a new curriculum is expanded
eagerly so the user can start; the rest expand lazily.

### GET /api/curriculums

- Returns curriculums for the authenticated user, ordered newest-first. Each
  item includes `course_count`, `completed_courses`, `module_count`,
  `completed_modules`, and `progress` (0–100).

### POST /api/curriculums

- Request:

```
{
  "topic": "string",
  "mode": "learn",
  "courses": [
    { "index": 0, "title": "string", "outcome": "string" }
  ]
}
```

- Response (status 201) includes the persisted curriculum and the eagerly
  generated first roadmap:

```
{
  "curriculum_id": "string",
  "topic": "string",
  "mode": "learn",
  "courses": [
    {
      "id": "string",
      "index": 0,
      "title": "string",
      "outcome": "string",
      "status": "not_started|in_progress|completed",
      "expanded": true,
      "roadmap_id": "string|null",
      "module_count": 0,
      "completed_modules": 0,
      "progress": 0
    }
  ],
  "first_course_id": "string",
  "first_roadmap": { /* same shape as roadmap detail */ },
  "course_count": 0,
  "completed_courses": 0,
  "module_count": 0,
  "completed_modules": 0,
  "progress": 0
}
```

### GET /api/curriculums/{curriculum_id}

- Returns curriculum detail with all course stubs and rollup progress.

### POST /api/curriculums/{curriculum_id}/courses/{course_id}/expand

- Idempotently generates (or returns) the `Roadmap` for the given course.
- Response:

```
{
  "curriculum_id": "string",
  "course_id": "string",
  "roadmap": { /* same shape as roadmap detail */ }
}
```

## Lessons

### GET /api/lessons/{lesson_id}

- Query params: `module_topic` (string), `mode` (`learn` | `solve`, default `learn`).
- Returns micro-theory and question set (no `answer_key` in the payload).

### POST /api/lessons/{lesson_id}/answer

- Request:

```
{
  "question_id": "string",
  "answer": "string"
}
```

- Response:

```
{
  "correct": true,
  "next_prompt": "string",
  "progress": {
    "xp": 0,
    "stars_remaining": 0,
    "status": "in_progress"
  }
}
```

### POST /api/lessons/{lesson_id}/hint

- Request:

```
{
  "question_id": "string",
  "hint_level": 1
}
```

- Response:

```
{
  "hint": "string",
  "stars_remaining": 0
}
```

## Dashboard

### GET /api/dashboard

- Returns lesson counts, streak info, and progress summaries for the authenticated user.
- Auth: `Authorization: Bearer <user_id>` header. Falls back to `anonymous` when absent.

- Response:

```
{
  "total_xp": 0,
  "total_stars": 0,
  "lessons_completed": 0,
  "lessons_in_progress": 0,
  "current_streak": 0,
  "longest_streak": 0,
  "recent_activity": [
    {
      "lesson_id": "string",
      "status": "not_started|in_progress|completed",
      "xp_earned": 0,
      "stars_earned": 0,
      "updated_at": "ISO 8601 timestamp"
    }
  ],
  "roadmaps": [
    {
      "id": "string",
      "kind": "roadmap",
      "topic": "string",
      "mode": "learn|solve",
      "module_count": 0,
      "completed_modules": 0,
      "created_at": "ISO 8601 timestamp"
    }
  ],
  "curriculums": [
    {
      "id": "string",
      "kind": "curriculum",
      "topic": "string",
      "mode": "learn",
      "course_count": 0,
      "completed_courses": 0,
      "module_count": 0,
      "completed_modules": 0,
      "created_at": "ISO 8601 timestamp"
    }
  ]
}
```

- `recent_activity` is capped to the 10 most recent entries, sorted newest-first.
- `current_streak` counts consecutive days with at least one lesson interaction.
- XP and stars are aggregated across all lesson_states for the user.

## Progress (internal — used by lesson answer flow)

### Progress_Updater skill

- Called internally after each `POST /api/lessons/{lesson_id}/answer`.
- Input: `correctness`, `hint_usage`, `timing_seconds`, `current_xp`, `current_status`, `answered_count`, `total_questions`.
- Output: `xp_earned`, `total_xp`, `stars_remaining`, `status`, `correctness`.
- This is a LOCAL (deterministic) skill — it does not call OpenRouter.
- State transitions are safety-checked: `completed` lessons cannot regress to `in_progress`.
